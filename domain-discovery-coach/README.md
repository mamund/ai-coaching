## 📖 Overview

The **Domain Discovery Coach** helps you explore a domain *before* you design anything.
It’s the first step in the AI-Coaching ecosystem — the stage where you learn how the real system behaves by observing **events**, **actors**, and **artifacts** in motion.

This coach encourages **seeing before shaping** — recording how work *actually happens* so that later coaches (Persona, API Story, Vocabulary) can build on truth, not assumption.

> “You can’t model what you can’t see. Discovery is about perceiving the system in motion.”

---

## 🎯 Purpose

Guide API and system designers through **reality capture** — the observational, pre-design phase where insights emerge from field study, interviews, or team discussion.
The outputs form the foundation for every subsequent artifact in the Narrative-First method.

---

## ⚙️ How It Works

### 🧩 Inputs

* Observations or interviews
* Logs, forms, or workflow artifacts
* Early goals or mission statements

### 🔄 Process

1. **Trigger Event** – What starts activity?
2. **Elicit & Observe** – Who acts and what happens next?
3. **Identify Events & Actions** – What changes or decisions occur?
4. **Cluster & Sequence** – How do these actions form a process?
5. **Name Participants** – Who performs or supports each step?
6. **Capture Examples** – “Given / When / Then” stories.
7. **Reflect on Success** – What signals that things are working?

### 📦 Outputs

* `discovery-summary.md` (auto-generated every session)
* Optional **Discovery ZIP Archive** containing:

  * `discovery-brief.md`
  * `event-inventory.md`
  * `roles-and-actors.md`
  * `example-table.md`
  * `domain-terms-seed.md`
  * `signals-of-success.md`
  * `opportunities-and-affordances.md`

---

## 🧠 When to Use

* At project kickoff, before Personas or API Stories exist
* When workflows are unclear or undocumented
* When teams need a shared understanding of “how things work today”

Short sessions (30–60 minutes) are ideal.
Each discovery run covers one domain or subsystem.

---

## 💬 Sample Prompts

* “Describe what usually triggers work in this domain.”
* “Who’s involved, and what do they do next?”
* “What artifacts record or support these actions?”
* “When everything goes right, how can you tell?”

---

## 🧩 Example Excerpt

> **Domain:** Shared Tool Shed
> **Trigger:** Neighbor borrows a tool
> **Events:** Borrow → Return → Repair loop
> **Actors:** Neighbor, Shed Supervisor, Repair Service
> **Artifacts:** Borrowing Slip, Tool List
> **Signals of Success:** Tools available, repairs timely, community satisfied

Outputs from this example feed directly into the Persona, API Story, and Vocabulary Coaches.

---

## 🧩 Ethical Use & Safety

* Do **not** include private, personal, or proprietary information.
* Use anonymized or synthetic examples.
* Treat all sessions as observable.
* Remember: these tools **augment human judgment**, not replace it.

---

## 💡 Tips for Beginners

* Focus on *what is*, not *what should be*.
* Capture reality in plain language.
* Highlight surprises, contradictions, or unknowns — they mark discovery.
